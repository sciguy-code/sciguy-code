## what i'm actually doing

**systems & low-level work**

i spend a lot of time with c/c++ trying to understand what's actually happening under the hood. this means working with pointers, memory layout, object semantics—the stuff that matters when you're trying to write efficient, correct code.

i implement data structures and algorithms from scratch rather than using libraries. not because libraries are bad, but because implementing things yourself forces you to think about edge cases, time complexity, and trade-offs. you learn more when you have to make it work yourself.

i also run small experiments to see how different approaches behave—allocation patterns, cache performance, that sort of thing. sometimes the "obvious" solution isn't the fastest one, and the only way to know is to measure.

working through os fundamentals too: processes, threads, basic synchronization. starting simple, staying focused on understanding the internals rather than building flashy demos.

**backend development**

i use **fastapi** for backend work when i need to build actual services. clean request/response contracts, proper validation, straightforward routing. nothing overengineered—just code that does what it's supposed to do.

some basic database work where it makes sense. the goal is always correctness first, then performance if it matters.

i treat frameworks as tools, not magic. understanding what they're doing behind the scenes is half the point.

planning to pick up **go** next. concurrency model looks interesting, and it seems like a good fit for systems-oriented backend work.

**problem solving**

algorithmic problems, competitive programming problems—i work through these regularly. not for the sake of grinding problems, but because they're a good way to practice reasoning through constraints and efficiency. every problem is a small puzzle.
